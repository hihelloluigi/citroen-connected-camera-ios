# Running the project

How to set up, run, test, and build Citroën Connected Camera locally.

## Prerequisites

- macOS with **Xcode 16+** (iOS 17 SDK).
- [Homebrew](https://brew.sh).
- [XcodeGen](https://github.com/yonaskolb/XcodeGen): `brew install xcodegen`
- [SwiftLint](https://github.com/realm/SwiftLint): `brew install swiftlint`

## First-time setup

```sh
cp Config/Secrets.example.xcconfig Config/Secrets.xcconfig
```

Open `Config/Secrets.xcconfig` and fill in `DEVELOPMENT_TEAM` with your Apple Developer Team ID.
This is only needed to run on a physical device (code signing is disabled for simulator builds);
you can leave it blank to build and run in the Simulator. `Config/Secrets.xcconfig` is git-ignored,
so your team ID never gets committed.

```sh
xcodegen generate
open CitroenConnectedCamera.xcodeproj
```

`CitroenConnectedCamera.xcodeproj` is generated by XcodeGen from `project.yml` and is git-ignored —
it does not exist until you run `xcodegen generate`, and it isn't tracked in version control.
Whenever you add, remove, or move source files, rerun `xcodegen generate` to regenerate the
project so Xcode picks up the change.

## Running the app

In Xcode, pick the **CitroenConnectedCamera** scheme and an iOS 17+ simulator or device, then Run.
The app launches into the onboarding flow (welcome → permissions → connect → set password →
reconnect), then hands off to the gallery.

> [!IMPORTANT]
> The onboarding "connect to Wi-Fi" step and the gallery talk to the physical camera over its own
> Wi-Fi access point at `192.168.0.1`. A simulator has no way to join that network, so it cannot
> reach the camera. On a simulator you can see the UI up through the "connect to Wi‑Fi" step, but
> to exercise the full flow (connect, set password, reconnect, browse/download/delete media) you
> must run on a **physical device joined to the camera's Wi-Fi network**.

## Permissions & entitlements

- **Access WiFi Information** entitlement (`com.apple.developer.networking.wifi-info`, in
  `App/App/CitroenConnectedCamera.entitlements`) — lets the app read the joined Wi-Fi network's
  SSID to confirm you're connected to the camera.
- **Info.plist usage strings** (`App/App/Info.plist`):
  - `NSLocalNetworkUsageDescription` — Local Network access, needed to reach the camera on your
    Wi-Fi.
  - `NSLocationWhenInUseUsageDescription` — Location (When-In-Use), needed to read the current
    Wi-Fi SSID on iOS.
  - `NSPhotoLibraryAddUsageDescription` — Photo Library add-only access, needed to save downloaded
    photos/videos to your library.
- **ATS exception**: `NSAppTransportSecurity` sets `NSAllowsLocalNetworking` to allow cleartext
  HTTP to local-network addresses, since the camera at `192.168.0.1` serves plain HTTP with no
  TLS.

On device, the OS prompts for Local Network access and (optionally — Location is not required to
use the app) Location during onboarding. Denying Local Network access blocks the app from ever
reaching the camera; denying Location only means the app can't display the confirmed SSID.

## Running the tests

The three Swift package suites run on macOS with no simulator and no camera required:

```sh
cd Packages/VIRBKit && swift test
cd Packages/AppCore && swift test
cd Packages/CoreUI  && swift test
```

These cover the VIRBKit camera client/transport (against mocked `URLProtocol` fixtures), AppCore's
routing/onboarding/gallery logic, and CoreUI's design tokens. The live OS/camera wrappers in the
app target (keychain, location permissions, Wi-Fi info, gallery service, photo library saving)
have no unit tests — they're verified by build success and manual testing on a physical device
instead. See the [architecture doc's testing section](architecture.md) for
details.

## Building & linting

```sh
xcodegen generate
xcodebuild -project CitroenConnectedCamera.xcodeproj -scheme CitroenConnectedCamera \
  -destination 'generic/platform=iOS Simulator' build
swiftlint --strict --config .swiftlint.yml
```

SwiftLint must report 0 violations. `force_unwrapping` and `force_try` are both configured as
errors (see `.swiftlint.yml`), so `--strict` fails the build on any force-unwrap or force-try
introduced anywhere under `Packages/VIRBKit/Sources`, `Packages/AppCore/Sources`,
`Packages/CoreUI/Sources`, or `App/App`.
